= VMs können mit dem NetApp Console Backup and Recovery Service gesichert und wiederhergestellt werden
:hardbreaks:
:allow-uri-read: 
:nofooter: 
:icons: font
:linkattrs: 
:imagesdir: ../media/


[role="lead"]
VMs in Red Hat OpenShift Virtualization können mit der NetApp Console gesichert und wiederhergestellt werden. Dieses Verfahren umfasst das Erstellen und Verwalten von Backup- und Restore-Vorgängen über den Backup and Recovery Service über die NetApp Console, wobei ein Azure Blob Storage Container als Sicherungsziel verwendet wird.

NetApp Console ermöglicht die zentrale Verwaltung von Speicher- und Datendiensten in lokalen und Cloud-Umgebungen. Es stehen einheitliche Kontrolle, einfache Bedienung und sicheres Management zur Verfügung. Mehrere Datendienste und Verwaltungsfunktionen sind über die Benutzeroberfläche unter link:https://console.netapp.com["console.netapp.com"^] zugänglich. Vollständige Informationen zu NetApp Console finden sich unter link:https://docs.netapp.com/us-en/console-family/["NetApp Console-Dokumentation"].

Dieser Abschnitt konzentriert sich auf den NetApp Backup and Recovery-Datendienst für Kubernetes-Workloads, insbesondere OpenShift virtuelle Maschinen. Der NetApp Backup and Recovery Service ermöglicht das Erkennen, Verwalten, Erstellen und Verknüpfen von Schutzrichtlinien mit Ihren Kubernetes-Anwendungen und VMs über eine einzige Schnittstelle. Eine vollständige Anleitung befindet sich unter link:https://docs.netapp.com/us-en/data-services-backup-recovery/br-use-kubernetes-protect-overview.html["NetApp Backup and Recovery Dokumentation"].



[[backup-and-recovery-workflow]]
== Backup und Recovery Workflow

. *Stellen Sie sicher, dass Sie einen Console Agent haben*
+
Stellen Sie sicher, dass in der Umgebung, in der Ihr OpenShift-Cluster ausgeführt wird, ein Konsolenagent bereitgestellt ist. Einzelheiten zur Installation eines Konsolenagenten sind in der link:https://docs.netapp.com/us-en/console-setup-admin/concept-agent-install-options.html#agent-console-deploy["NetApp Console Setup-Dokumentation"] enthalten.

+
.Beispiel anzeigen
[%collapsible]
====
In der NetApp Console auf „Agent bereitstellen“ klicken, wie im Screenshot gezeigt, und den Agenten in der Umgebung bereitstellen, in der der OpenShift Cluster ausgeführt wird.

In diesem Beispiel wurde die Option On-Premises gewählt. Die URL des Agenteninstallationsprogramms wird kopiert, mit dem Befehl curl wird das Agenteninstallationsprogramm auf dem unter vSphere laufenden Ubuntu-System heruntergeladen und das Installationsprogramm wird ausgeführt, um den Console Agent bereitzustellen.

Registrieren Sie ihn mithilfe der IP-Adresse des Agenten in der URL. Siehe link:https://docs.netapp.com/us-en/console-setup-admin/task-install-agent-on-prem-ova.html["die Dokumentation"] für ausführliche Anweisungen. Nach der Registrierung des Agenten ist dieser in der NetApp Console wie im untenstehenden Screenshot sichtbar.

. *Einrichten eines Azure Blob Storage Containers als Sicherungsziel für ein Sicherungs- und Wiederherstellungsszenario für die OpenShift VMs.*
+
In diesem Beispiel wird Azure Blob Storage als Backup-Ziel verwendet. AWS S3, Google Cloud Storage, StorageGRID oder ONTAP Object Storage können ebenfalls als Backup-Ziele verwendet werden.

+
Weitere Einzelheiten sind unter link:https://docs.netapp.com/us-en/data-services-backup-recovery/br-start-setup.html#prepare-the-backup-destination["NetApp Console-Dokumentation"] zu finden.

+
Um den Azure Blob Storage Container als Sicherungsziel in der NetApp Console zu erkennen, werden die folgenden Informationen aus dem Azure-Portal benötigt.

+
[listing]
----
Application (Client) ID:
Client Secret:
Directory (tenant) ID:
----
+
Um diese Werte zu erhalten, sind im Azure-Portal die folgenden Schritte erforderlich.

+
.. Einen Blob Storage-Container erstellen.
.. Erstellen Sie eine Enterprise-Anwendung und weisen Sie dieser Anwendung die erforderlichen Zugriffsrechte für den Blob Storage Container zu. Die Anwendungs-ID, die Verzeichnis-ID und das Client Secret dieser Anwendung werden benötigt, um in der NetApp Console den Azure Blob Storage Container als Sicherungsziel zu erkennen.
+
.Beispiel anzeigen

.... Eine Ressourcengruppe im Abonnement erstellen.

. *In der NetApp Console das OpenShift Cluster entdecken.*
+
.Beispiel anzeigen
[%collapsible]
====
Für diesen Schritt muss der OpenShift Cluster ausgeführt werden und der Console Agent in derselben Umgebung bereitgestellt sein. Die folgenden Schritte ermöglichen die Erkennung des OpenShift Clusters in der NetApp Console.

.. In der NetApp Console *Inventar > Kubernetes* auswählen und die Details des OpenShift Clusters angeben.
.. Wählen Sie den Agenten aus, der in derselben Umgebung bereitgestellt ist, in der sich der OpenShift Cluster befindet.
.. Kopieren Sie die bereitgestellten Befehle, um Trident Protect zu installieren, und führen Sie sie von einem Rechner aus, der zum Ausführen von  `oc`Befehlen auf Ihrem OpenShift Cluster verwendet wird.
.. Nach erfolgreicher Installation von Trident Protect auf *Erkennen* klicken. Die NetApp Console erkennt den OpenShift Cluster und alle seine Ressourcen über den Agenten und stellt sie in der Benutzeroberfläche dar.

. *In der NetApp Console eine Anwendung für die VMs erstellen und mit Backups schützen*
+
.Beispiel anzeigen
[%collapsible]
====
In diesem Schritt wird in der NetApp Console eine Anwendung für die VMs erstellt, die geschützt werden sollen. Es kann eine Anwendung für eine einzelne VM oder mehrere VMs erstellt werden. Es besteht auch die Möglichkeit, Namespaces zu verwenden, um eine Anwendung zu erstellen, die alle VMs in einem Namespace im OpenShift Cluster schützt.

Sie benötigen außerdem eine Datensicherungsstrategie, die Sie der Anwendung zuordnen. Es kann eine neue Datensicherungsstrategie erstellt oder eine vorhandene verwendet werden. In diesem Beispiel wird eine neue Datensicherungsstrategie zusammen mit der Anwendung erstellt und ihr zugeordnet. Backups für die Anwendung werden basierend auf der zugeordneten Datensicherungsstrategie erstellt. Die Anzahl der aufbewahrten Backups richtet sich nach der in der Datensicherungsstrategie definierten Aufbewahrungsrichtlinie.

----
- apiGroups:
  - groupsnapshot.storage.k8s.io
  resources:
  - volumegroupsnapshots
  verbs:
  - get
  - list
  - watch
  - create
  - delete
  - update
  - patch
- apiGroups:
  - groupsnapshot.storage.k8s.io
  resources:
  - volumegroupsnapshotcontents
  - volumegroupsnapshotcontents/status
  verbs:
  - get
  - list
  - watch
  - create
  - delete
  - update
  - patch
- apiGroups:
  - groupsnapshot.storage.k8s.io
  resources:
  - volumegroupsnapshotclasses
  verbs:
  - get
  - list
  - watch

----
